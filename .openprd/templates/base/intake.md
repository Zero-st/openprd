# Intake 模板

> 默认先用业务和产品语言回答就可以，不需要一上来讲技术实现。

## 发现

- 我们要解决什么问题？
- 主要是给谁用的？他们会在什么场景下最先感受到价值？如果你愿意，也可以直接说它更偏面向个人消费者、面向企业服务，还是以 Agent 为主。
- 第一批最容易触达、最可能愿意给真实反馈的社区、渠道或人群是谁？
- 你为什么算这个社区里的“自己人”？他们平时主要在哪里聚集、为什么你现在就能触达？这是不是一个足够小、足够集中的切口？
- 第一版最小可用切片，最希望先让用户完成什么关键动作？
- 当前临时解决方式是什么？
- 这个问题到底有多痛？用户现在是不是已经在为更差的办法花时间、花钱，或者至少在强行 workaround？如果一句话描述客户和痛点，会怎么说？
- 如果先不做完整产品，能不能先靠手工服务、半自动流程或轻量交付把价值跑出来？
- 如果先手工交付，这份“手工作战卡”该怎么写：触发条件、关键步骤、用什么工具、单客大概花多久、谁来接力？
- 如果今天禁止你写复杂代码，只能用 spreadsheet、表单、Zapier/Notion/Airtable 这类现成工具，你会怎么把第一版跑起来？哪些东西现在明确先不 build？如果已经不得不做产品，也只自动化最重复的一步，先压成 forms / lists / CRUD 这种最轻骨架。
- 第一版到底只做哪一件事？它能不能被压成一个周末级 MVP、一个顾问式试跑，或一个更轻的验证动作？
- 你现在至少能不能点出 3 到 10 个具体的人、团队或客户样本？如果已经聊过，里面有多少人真的愿意付费、预约试跑、导入真实数据，或者给出同等级别的承诺？
- 第一批客户最现实的触达顺序是什么？先熟人、现有客户、社区，还是陌生线索？
- 从第一个客户开始准备怎么收费？什么价格既是 no-brainer，又不会因为免费而失真？
- 什么真实承诺最能证明这不是口头兴趣，比如付费、预约、提交真实素材、拉人进群或导入数据？
- 最低成本先验证哪一步最关键？如果验证结果一般，什么情况下应该先停，或者先换更轻的方式继续？
- 如果只拿到第一个客户，这条路怎么打平时间和交付成本？更接近 default alive 还是 default dead？
- 你准备在什么条件下才允许自己“产品化”或继续加功能？例如已经服务了 10+ 付费客户、需求已经稳定几周、手工步骤几乎不再变化。一旦真的开始做产品，默认也只自动化一小步，不直接做大平台。
- 销售和增长这条线准备怎么守纪律？是不是先熟人、再社区、再陌生线索；100 个付费客户前不把 launch 当获客策略；先花时间别先花钱；有自然增长前不急着投广告或招人。
- 如果结果一般，这个方向有多可逆？更像是在解决客户真问题，还是在满足我们自己的技术冲动？
- 这条路是否符合团队现在想坚持的价值观和做事方式？如果连续这样做 3 到 5 年，你还愿意住在这套业务里吗？
- 哪些内容这轮先不做？
- 哪些既有能力、流程、体验或业务结果不能被破坏？
- 成功后，用户会明显感受到什么变化？我们怎么判断它值得做成？
- 是否涉及账号身份、用户数据、多人协作、AI 调用、外部对接、收费或其他会带来业务风险和成本的行为？

## Freeze Check

- 产品类型是否已明确？
- 关键角色是否已明确？
- 第一版切片是否已明确？
- 非目标和保护项是否已明确？
- 是否已有成功指标？
- 若存在消耗型成本，是否已明确额度限制、滥用防护、监控报警和止损动作？
- 是否存在阻塞 freeze 的开放问题？

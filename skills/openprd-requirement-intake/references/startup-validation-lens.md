# 创业验证透镜

当 L2 需求、脑暴模式或 0 到 1 方向讨论还停留在“做什么功能”时，用这组问题把它收回到更真实的创业验证闭环。重点不是多补几个字段，而是把“怎么保持小、什么时候不要继续做大”变成默认判断链。

## 保持小的六道门

### 1. 社区门

先问是不是这个社区里的自己人，而不是先问功能点子够不够酷。

- 第一批最容易触达的社区、渠道或人群是谁？
- 你为什么算这个社区里的自己人？你理解他们的语言、价值观和工作方式吗？
- 你现在是否已经在这个社区里真实参与、贡献或交付，而不是只把它当作潜在市场？
- 他们主要在哪里聚集？你现在就能接触到他们吗？
- 你能不能说出至少 10 个具体的人、团队或试点对象？
- 这个切口是不是足够小、足够聚焦，能先在一个小圈层里做深？

如果连 10 个具体对象都说不出来，或你其实不属于这个社区，先不要急着进入方案设计。

### 2. 痛点门

先看现在怎么解决，再看值不值得替换。

- 他们现在主要靠什么替代方案、流程、人工 workaround 或更差的付费方案在解决？
- 这个问题到底有多痛？只是轻微不爽，还是高频、卡业务、卡收入、卡交付？
- 他们已经在为这个问题花什么时间、精力或金钱？
- 能不能用一句话说清“谁在什么场景下，被什么问题反复卡住”？
- 如果需要先教育用户“你其实有这个问题”，这通常是红旗。

当前 workaround 就是你的真实竞争对手；如果没有 workaround，往往说明痛点还不够真。

### 3. 最小执行门

先 processize，再 productize。

- 如果今天禁止你写复杂代码，只能用自己、电话、表单、spreadsheet、Zapier、Notion、Airtable 这类现成工具，你会怎么把价值跑出来？
- 先不做完整产品时，怎么靠手工服务、半自动流程或轻量交付把结果交出去？
- 这份手工作战卡怎么写：触发条件、关键步骤、工具、单客耗时和交接点分别是什么？
- 第一天到底只做哪一件事？一句话讲不清，就说明范围还没收够。
- 能不能压成周末级 MVP、顾问式试跑或更轻验证？
- 如果已经不得不开始做产品，能不能只自动化最重复的一步，并把第一版压成 forms / lists / CRUD 这种最轻骨架？
- 这一步是在服务今天已经存在的客户和重复需求，还是在为假想中的未来规模、未来角色和未来复杂度提前造壳？

这里的“手工作战卡”就是 `Magic Piece of Paper`。重点不是格式漂亮，而是让别人拿到也能继续交付。

### 4. 付费验证门

兴趣不算验证，交易或更强承诺才算。

- 你是否至少和 10 个潜在对象聊过，或已经能明确列出这 10 个样本？
- 这 10 个样本里，是否至少有大约 3 个愿意付费、预付、安排试点，或给出更强的真实承诺？
- 什么真实承诺最能证明这不是口头兴趣？例如愿意预约、导入真实数据、提交真实素材、拉同事一起试、接受明确上线窗口。
- 从第一个客户开始准备怎么收费？什么价格既是 no-brainer，又不会因为免费而失真？
- 第一个客户如何覆盖时间和交付成本，让这条路更接近 default alive？

“早收费”不是为了立刻赚钱，而是避免 zero price effect。免费可以短时例外，但不能长期拿免费当真需求证明。

### 5. 产品化门

不是“有点反馈了就继续加功能”，而是“达到门槛才允许产品化”。

- 在继续产品化前，要不要先满足这些条件：已经服务过 10+ 付费客户、需求开始重复、手工作战卡几周没大改、你花在交付上的时间开始明显高于找客户？
- 如果这些条件还不满足，当前最合理的动作是不是继续手工、继续 no-code、继续顾问式试跑，而不是扩成功能列表？
- 现在明确先不 build 的东西有哪些？
- 哪一步最值得先自动化？为什么是它，而不是整个系统一起上？
- 就算开始做产品，第一版是否也只自动化这一小步，而不是马上做成平台、工作台或全流程系统？
- 如果现在继续做大会产生哪些不可逆决策？例如重招聘、长期绑定、复杂权限、重平台化或高维护承诺。

默认先自动化最费时、最稳定、最重复的那一步，而不是把整套系统一次性做完。

### 6. 增长门

先卖，再 launch；先花时间，再花钱。

- 第一批客户最现实的触达顺序是什么？先熟人、现有客户、社区，还是陌生线索？
- 在有足够付费基础前，是否应该把 launch 当成庆祝而不是获客策略？
- 在没有自然增长和明确客户画像前，是否应该先花时间做内容、访谈、私聊和手动销售，而不是先投广告或招人？
- 什么情况下才允许进入更主动的营销、付费增长或团队扩张？
- 这条路怎样才能先活下来，而不是先把成本、复杂度和维护负担做爆？

`slavingia/skills` 的强约束是：没有真实付费基础前，不要把 marketing、launch、ads 和规模化错当成解决验证问题的手段。

## 常见红旗

- 目标用户太泛，像“所有创业者”“所有做内容的人”“所有用 AI 的团队”。
- 你并不在这个社区里，只是觉得这个市场可能很大。
- 需要先教育别人“你有这个问题”，而不是顺着现有抱怨或现有 workaround 切入。
- 第一反应是做完整产品，而不是先做手工服务、表单、spreadsheet 或 no-code。
- 明明还没付费验证，就开始讨论规模化架构、复杂权限、自动化编排或营销投放。
- 还没拿到稳定重复需求，就先往“多功能平台”“一站式工作台”“全流程闭环”扩。
- 现在服务的还是假想中的未来客户，而不是今天已经能找到、已经在痛的那批人。
- 还没跑顺最小成交链，就先做重招聘、长期绑定或其他难回头的决策。

## 使用方式

- 不要机械逐条盘问；优先把这六道门融进 requirement 摘要、脑暴卡片和追问顺序里。
- 如果用户已经明确提供其中几项，就直接吸收，不要重复确认。
- 对内部工具、团队流程或 Agent 项目，也照样适用。
  这里的“社区 / 种子用户”可以是维护者、运营、交付、内部试点团队或最先受影响的一线角色。
- requirement 摘要和 brainstorm 默认至少要覆盖：
  - `communityFit`：为什么算自己人、在哪里触达、为什么是小而可达的切口
  - `painEvidence`：当前 workaround、痛点强度、已付出的时间或金钱
  - `paymentProof`：10 个样本、3/10 付费意愿或更强承诺
  - `smallestExecution`：spreadsheet / 表单 / no-code / 手工桥接，以及明确不 build 的部分
  - `productizeGate`：达到什么条件才继续产品化
  - `growthDiscipline`：熟人到陌生线索的顺序、何时不该 launch、先花时间别先花钱
- “default alive”不是财务口号，而是提醒 Agent 先问清楚：这件事在验证阶段怎样不把成本、复杂度和维护负担一下子做爆。
- “minimalist review” 不只看市场大不大，还要看这条路是不是更简单、更可逆、更贴近客户真问题，并且不违背团队想坚持的价值观；更进一步，还要问这是不是你愿意长期住进去、不会反过来绑住自己的业务形态。
